# 课程讲稿：第六节 - 同时添加多个功能：Git Worktrees 与自定义命令（全文翻译）

你可以开启多个 Claude Code 会话来并行开发多个功能。为了管理这些会话并确保不会覆盖同一个文件，你可以使用 Git worktrees。现在，让我们添加 Git worktrees，同时为聊天机器人添加三个并行功能。

回到 Claude 界面。正如我们之前看到的，Claude Code 内置了相当多的斜杠命令（slash commands），但你也可以创建自己的自定义命令。

**要创建自定义命令，请在 `.claude` 文件夹内新建一个名为 `commands` 的文件夹。**在里面，**我们将创建一个 Markdown 文件，文件名就是你想要的命令名**。我在这里要创建的命令叫作 `implement-feature`。所以我创建了一个名为 `implement-feature.md` 的文件。

在这个文件中，你可以放入任何你想要的内容。但我想向你展示一个非常特别的功能：**如果你想向自定义命令传递任何参数，可以使用 `$ARGUMENTS` 变量来引用它**。

我在这里的要求是：当使用此命令时，我会指明你正在实现一个新功能。用户可以指定该功能是什么，然后我希望确保 Claude Code 清楚地知道只针对前端功能执行此操作，并将更改写入一个名为 `frontend-changes.md` 的文件中。

你可以想象，自定义命令有很多不同的用例，比如特定的测试运行方式、运行文件的方式等。**需要注意的是，你放在这里的内容不会像 `CLAUDE.md` 那样自动添加到上下文中。**因此，**如果你希望某些内容应用于你开启的每一个 Claude Code 实例，请使用 `CLAUDE.md` 文件**。但如果你只是有一些可能在不同对话中选择性使用的特定命令，这里就是一个非常好的存放地。

> 意思是只有主动调用才会生效。

我将在开始讨论 worktrees 时使用这个自定义命令。但在那之前，让我们退出 Claude Code 再重新进入，验证一下是否能看到该自定义命令。

瞧，我看到了 `implement feature`，其描述正是来自 Markdown 文件的第一部分。现在我已经添加了这个自定义命令，让我们把它添加并提交。我可以在命令行中执行此操作，但我打算让 Claude 帮我完成。要求：“添加并提交已做的更改。” 这样很好，因为 Claude Code 可以生成一个精练且带有描述性信息的提交消息。

我们可以看到它正在添加 `.claude` 文件夹并提交。现在这已添加到存储库中。由于我们之前已经授予了权限，所以无需再次响应确认。**如果你好奇这些权限信息存在哪里，在 `.claude` 文件夹内有一个 `settings.local.json` 文件。在这个文件中，我们可以指定已允许的命令，这样就不需要每次都确认。**

同样，正如你在这里看到的，当我们使用 Playwright 时，我们也给予了权限。如果你想添加自己的权限，可以轻松地在这个文件中修改，或者使用便捷的 `/permissions` 命令。

现在自定义命令已经设置好了，让我们聊聊如何与 Claude Code 并行工作。与其只是打开多个终端窗口并直接在这个代码库上操作，我们将使用 Git 来确保在拥有多个 Claude Code 实例时不会覆盖现有文件。

如果不加管理，我可能会有两个不同的 Claude Code 实例在操作同一个文件，这会导致覆盖、产生 Bug 以及相当大的混乱。幸运的是，**Git 有一个非常棒的功能叫作 worktrees**。Worktrees 允许我基本上创建代码库的副本，在隔离环境中操作，最后再将它们合并在一起。事实上，我还可以利用 Claude 来协助合并和管理这些 worktrees。

要开始使用 worktrees，我首先创建一个名为 `.trees` 的文件夹。在这个文件夹里，我将添加几个 worktrees。**使用 `git worktree add` 命令，指定文件夹以及我想要的 worktree 名称。**我们将第一个命名为 `ui_feature`，第二个命名为 `testing_feature`，最后第三个叫作 `quality_feature`。

```bash
# 创建一个名为 .trees 的文件夹
mkdir .trees

# 创建 ui_feature worktree - 基于main分支
git worktree add .trees/ui_feature main

# 创建 testing_feature worktree - 基于main分支
git worktree add .trees/testing_feature main

# 创建 quality_feature worktree - 基于main分支
git worktree add .trees/quality_feature main

# 确认已创建所有 worktrees
git branch -a
```

> 属于 Git：它是 Git 在 2.5 版本（2015年发布）中引入的一个内置功能。
>
> 创建了三个不同的新分支，并且每个分支都关联了一个独立的本地文件夹（工作区）

为了确认已创建所有这些，我可以查看一下，可以看到我当前在 `main` 分支上，并且创建了三个独立的 worktrees。为了在这些环境中正确设置，我将打开 `.trees` 文件夹，并为每一个 worktree 开启一个终端。

我先把 `ui_feature` 移到这边，接着把 `testing_feature` 放在旁边，最后把 `quality_feature` 放在上面。为了腾出更多空间，我们把窗口调小一点，并隐藏资源管理器。现在我们有了三个专用的终端窗口。

现在，在每个环境中都打开 Claude。我们现在可以并行运行 Claude Code，并确保如果修改了相同的文件，我们不会互相覆盖，稍后合并这些 worktrees 时再进行修复。

我将使用 `implement-feature` 命令，然后输入我想要的一个特定功能。该功能是：添加一个切换按钮来在深色和浅色主题之间切换。我们将开始创建按钮、定位，并确保可以导航。

在这个运行的同时，我可以移动到另一个 worktree。在第二个 worktree 中，让我们考虑一下测试框架。我输入一个提示词，要求增强现有的测试框架，并为 FastAPI 端点添加额外的测试。

在那个运行的同时，让我们在第三个窗口为开发流添加一些核心的代码质量工具。我们可以看到各个环境都在请求更改。让我们在这些环境之间切换并确认更改。测试正在编写中，同时正在添加用于分析代码库结构和格式化的额外开发依赖项。

我们会看到正在修改 `pyproject.toml` 文件。我们可以并行工作，确保这些更改不会被代码库的其他部分覆盖。如果碰巧对相似的文件进行了修改，我们可以确保修复那些合并冲突，而且我们可以利用 Claude 来完成。

前端更改正在写入，我们在添加更多测试，还在添加格式化工具和用于质量检查的开发脚本。看起来一切都在按预期进行。现在，像之前一样使用自定义命令添加浅色主题支持，并将其记录到变更日志中。

在运行的同时，我可以检查其他脚本。确认我想要运行它们，如果有问题，Claude Code 会根据需要修复。似乎有一些 Claude 不知道的脚本，所以我们要么安装依赖项，要么确保按预期运行。

你可以看到，我们从代码质量检查开始，编写了 API 端点测试，并添加了更多前端功能。一旦全部完成，我们可以前往浏览器确认，但在那之前，让我们汇总所有这些独立的更改。

我们看到 `pyproject.toml` 文件在两个不同的 worktrees 中都被修改了，所以合并这些部分时可能会有冲突。既然我们对每个更改都感觉良好，让我们通过描述性消息进行添加和提交。在每个 worktree 上都执行此操作。由于我们要合并这些提交，我们需要清晰的提交消息来理解每个工作区做了什么。

我们完成了 UI 增强、测试编写和质量工具添加。如果你发现自己经常写“添加并提交并附带描述性消息”之类的提示词，这也可以是自定义命令的一个很好的用例，我们可以在其中指定公司遵循的 Git 最佳实践。

现在我已经完成了这三个工作区的提交，可以回到 `main` 分支进行合并了。我关闭这些终端环境，回到主终端。

现在我要求 Claude：**使用 `git merge` 命令合并 `.trees` 文件夹中的所有 worktrees，并修复可能存在的任何冲突。**让 Claude Code 合并这些特定的 trees 并确保它们按预期工作。

我们可以看到有三个可用的 worktrees。我们开始逐一合并。看起来 `testing_feature` 没有冲突。接着合并 `ui_feature`。再次强调，我们可以自己写这些命令，但 Claude Code 知道该怎么做。

> 每一个命令都是模糊的，意味着会花费token。

最后合并 `quality_feature` 分支。在这里，我们可以看到在某个特定文件中存在冲突。我们将让 Claude Code 分析这些冲突并完成合并。当你不想每次都手动处理细小的合并冲突时，这非常有用。

拥有测试也非常有价值，可以确保合并完成后运行测试，代码库仍能按预期工作。它正在继续合并并提交已解决冲突的所有更改。如果愿意，我之后可以要求它移除这些 worktrees。

我做一个快速测试，确保文件都在，且合并配置正确。我们回到浏览器看看前端更改。Claude Code 已经完成了：添加了必要的依赖、修改了 `pyproject.toml`、添加了测试、实现了深浅色主题，还实现了 Black 代码格式化工具以确保代码符合特定格式。

我们修复了所有冲突。回到浏览器，我看到了这个漂亮的主题，当我点击切换时，可以看到浅色和深色主题。

虽然可能还有一些细节需要调整，但我已经能够跨技术栈的所有部分进行编辑，甚至处理了 Linting 和 DevOps 方面的工作，而这一切都没有发生覆盖或产生令人头疼的冲突——这全靠 Git worktrees 的力量。
