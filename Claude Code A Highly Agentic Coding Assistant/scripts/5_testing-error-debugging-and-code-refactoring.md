# 课程讲稿：第五节 - 测试、调试与代码重构（全文翻译）

目前代码库中缺少评估机器人 RAG 流水的测试。让我们实现这些测试，用它们来调试机器人处理查询时出现的一个错误，并最终重构机器人处理工具调用的方式。让我们开始吧。

到目前为止，我们已经了解了如何快速熟悉代码库并实现功能。现在，假设距离我们上次处理这个应用已经过去了一段时间。我们回来想重新开始使用它。让我们回到应用，询问关于第五课内容的详细信息。

我们预想会得到信息反馈，但现在出问题了。你可能想直接复制这个错误扔给 Claude，或者截个图期望它能解决问题。但我们要采取一种略有不同的方法。我们不会只是寻求答案或让 Claude 盲目尝试，而是要采取更系统的方法。

我们知道应用出错了，但也知道目前没有足够的测试来程序化地验证这一点。所以，我们将输入一段提示词，不仅描述错误，还指定需要编写测试的地方。

回到代码库，这个错误可能源于多个 Python 文件。在 `AIGenerator` 中，我们处理与 Anthropic API 的交互，Prompt 或逻辑可能存在问题。在 `rag_system.py` 中，它是 RAG 全流程的编排者。在 `search_tools` 中，我们定义了底层的工具。

所以，我们首先要求 Claude 为这些特定文件编写测试。然后让 Claude 运行这些测试，通过这种方式验证哪里坏了，并进行修复。这种方法的强大之处在于：**通过测试先行，我们可以建立稳固的基础，准确理解为什么会失败**。

打开 Claude 输入提示词。由于这个任务更有挑战性，我们要开启 Claude 的**扩展思维（Extended Thinking）**能力，分配更多 Token 用于思考过程。我们可以观察这个思考过程，如果发现方向不对，随时可以中断它。

> 实际就是在最后输入 `Think a lot` 这句话。

我们会开启**计划模式（Plan Mode）**，因为我们希望在 Claude 开始写测试之前，先确保它理解了任务并获得我们的批准。在读取必要文件时，我们可以看到 Claude 的思考逻辑，包括它需要检查什么以及准备怎么做。

来看看 Claude 的计划。它似乎发现了一个配置问题，可能是复杂的工具调用失败点，或者是组件间的错误传播受限（导致错误被某处截获了）。它随后建议基于现有文件使用 `pytest` 框架构建结构，并使用 Mock 模拟 ChromaDB。这看起来是个好的开始。

它开始创建测试文件夹并编写测试。我们还会使用 `UV` 来安装必要的依赖，确保 `pytest` 运行正常。

在编写这些测试时，我们可以浏览生成代码。这里有课程搜索工具的测试、Fixture 和 Mock。这看起来很棒，AI 生成器、RAG 系统和向量数据库的测试也一应俱全。它似乎已经识别到了一个关键的配置问题，虽然写测试可能已经顺带解决了 Bug，但这些测试能给我们完整的保证：这就是问题所在，且修复是正确的。

> 这个课程全程 token 用量飞快，目前就不尝试了。

创建完测试后，使用 `UV` 运行它们。根据它的发现，问题似乎出在 `MAX_RESULTS` 或向量搜索返回的块数量上。查看 `MAX_RESULTS` 后确认了问题：由于某种原因，它被设为了 0。一旦我们修改这个值，不仅测试会通过，我们也能得到完整的结果。

Claude Code 正在收尾，它提供了一份详尽的发现总结。它完成了调试，找到了关键问题，并创建了一些测试和测试基础设施。

回到浏览器测试。开启新对话，再次询问同样的问题。现在不再报错，而是准确返回了课程信息。我们不仅修复了 Bug，还建立了一套强大的基础设施，确保后续修改不会在不知情的情况下破坏功能。

应用运行正常后，聊聊代码库的重构。在 `ai_generator.py` 中，我们目前规定每次查询最多只能调用一次搜索工具。虽然这对于简单查询有效，但当我们要进行复杂查询（如对比不同课程的大纲）时，需要不止一次的工具调用。我们需要一个能迭代处理所有必要工具或递归解决问题的环境。

如果你熟悉这里的代码，你会看到目前的类通过 SDK 直接与 Anthropic 模型对话，设置基础参数并生成响应。但在构建系统 Prompt 和消息时，没有迭代过程来实现多轮对话和多次工具调用。

让我们重构它。我有一段很长的提示词，我打算新建一个名为 `backend-tool-refactor.md` 的 Markdown 文件来存放它。在文件中，我要求 Claude 重构 `ai_generator.py` 以支持分轮次的多次调用。我描述了现状和期望的行为。给 Claude 尽可能多的信息是很有帮助的，特别是任务复杂时。我们还提供了一个示例流程（如搜索讨论相同主题的其他课程），让 Claude 明白为什么要多次调用。

注意，这里有几个不同的要求和提示，确保我们编写的是验证外部行为的测试，而不是关注内部状态细节。我还要求 Claude 思考多种方案，不要直接写代码，而是派出两个**子代理（Subagents）**来头脑风暴不同的选项。

> 这里用自然语言要求他运行两个平行的子代理。

我不确定最优的重构方式是什么，所以我给 Claude 机会让它并行思考多个选项。**运行该提示词，我会关闭“自动接受编辑”，以便逐一确认更改。**

我们可以看到两个并行的子代理正在使用 `task` 工具执行任务。两个方案出来了，由我们决定实现哪一个。它们并行操作，读取文件并想出两种不同的重构途径。

Claude 返回了建议：方案 A 较简单且向后兼容，方案 B 更全面但更复杂（支持多轮逻辑和不同的辅助方法）。从第一眼看，方案 A 更简洁。那我们就选方案 A 吧。开启计划模式，获取详细计划。

> 基本都是先做计划在修改。

我们要修改调用这些工具的方式。来看看具体的改动：更新方法签名以支持最大轮数，更新系统 Prompt，最核心的重构在于 `handle_tool_execution`。如果一切顺利，我们应该能看到：用户查询 -> 多次工具调用 -> 最终响应。且不影响其他部分。

现在看实现过程：修改现有方法，更新并运行测试。不需要切换到浏览器或找别人测试，在终端里就能搞定。

更新完系统 Prompt 和测试后，添加一个测试确保顺序工具调用正常。运行测试，全部通过。

回浏览器验证。首先询问某一节课的细节。我们不仅得到了信息，还得到了标题。特别之处在于：第一个工具调用只给出大纲，第二个工具调用才给出该课的细节和标题。这证明了多次调用生效了。

现在问个更难的：“还有其他课程涵盖了 MCP 课程第五课相同的主题吗？”为了回答这个问题，机器人需要多次调用工具：先获取 MCP 课程的信息和大纲，再搜索其他课程的大纲进行对比。它准确地回复说，目前没有其他课程涵盖构建 MCP 客户端的内容，这很准确。

在本课中，我们学习了如何使用 Claude Code 修复 Bug、在全流程中编写测试，并进行了重构以处理更复杂的查询。在下一课中，我们将通过运行多个 Claude 会话并使用 Git worktrees 来提升效率。回头见。
