# 课程讲义：第五节 - 测试、调试与代码重构

## 1. 系统化调试流程

面对应用报错，不要直接盲目修补，应遵循以下步骤：

1. **编写测试先行**：先写出能复现该 Bug 的测试用例（pytest）。
2. **定位核心组件**：识别错误可能源自 `AIGenerator`（Prompt 逻辑）、`RAG System`（编排逻辑）还是 `Search Tools`（工具定义）。
3. **运行测试验证**：通过测试失败点定位具体变量（如本课中的 `MAX_RESULTS` 被设为 0）。
4. **修复与回归**：修复后确保所有测试通过，建立长期保障。

## 2. 增强型思维 (Extended Thinking)

对于逻辑复杂、涉及多个文件耦合的任务：

- **触发方式**：在提示词中明确要求“请深入思考”、“详细分析”或使用专门的触发词。
- **作用**：Claude 会分配更多 Token 进行内部推理，并在输出前展示思考链。

## 3. 多方案决策与子代理 (Subagents)

当不确定最优重构方案时，利用并行思考：

- **技巧**：要求 Claude 派生两个子代理同时构思方案 A 和方案 B。
- **决策标准**：对比“实现简单/向后兼容”与“架构全面/长期维护性”之间的权衡。

## 4. 后端工具调用重构 (Multi-turn Tool Use)

- **现状**：许多简单的 RAG 实现每次查询仅支持一次工具调用。
- **目标实现**：通过重构 `handle_tool_execution`，实现“迭代”或“递归”式的工具使用。
- **应用场景**：处理复杂查询（如：“对比课程 A 和课程 B 的差异”），这需要先获取 A 的信息，再根据结果获取 B 的信息，最后进行综合。

## 5. 关键工具：UV

- **功能**：极速的 Python 包管理工具。
- **场景**：在调试过程中快速安装测试依赖（pytest, mocks 等），确保测试环境的一致性。

---

**核心心法**：优秀的 AI 辅助编程不是让模型直接给出最终答案，而是利用模型快速构建“测试-反馈-修复”的闭环。
